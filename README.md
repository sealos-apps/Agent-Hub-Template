# Agent Hub Template

Agent image templates for Sealos Agent Hub.

This repository defines a stable contract for building, publishing, and deploying agents in Agent Hub. Each agent can keep its own upstream installation and runtime behavior, while image builds, container entrypoints, deployment manifests, and template metadata stay consistent.

## Base Image

`base/` builds the shared Agent Hub Devbox base image used by all agent images.

The base image is self-contained in this repository. Required Devbox-compatible tooling is vendored under `base/tooling/`, so CI and release workflows do not need to checkout an external `devbox-runtime` submodule.

The base image intentionally keeps only high-value common dependencies:

- Node.js and npm
- Python and uv
- ripgrep
- minimal system utilities required by the shared runtime
- s6-based process entrypoint support

Agent-specific dependencies belong in each agent's own `install.sh`, not in the base image.

## Agent Directory Contract

Every agent directory must provide:

- `Dockerfile`
- `build.env`
- `install.sh`
- `entrypoint.sh`
- `template.yaml`
- `manifests/devbox.yaml.tmpl`
- `manifests/service.yaml.tmpl`
- `manifests/ingress.yaml.tmpl`
- `README.md`

Agent directories must not provide legacy runtime files such as:

- `config.sh`
- `config.json`
- `deploy.yaml`
- `bootstrap.sh`
- `healthcheck.sh`

Runtime configuration should come from environment variables, Kubernetes Secrets, ConfigMaps, or mounted files.

## Runtime Contract

All agents use the same startup chain:

```text
/init
  -> /opt/agent/entrypoint.sh
    -> /opt/agent/bin/start
      -> real upstream agent runtime
```

Rules:

- `ENTRYPOINT` is always `["/init", "/opt/agent/entrypoint.sh"]`.
- `CMD` is always `["start"]`.
- `entrypoint.sh` must stay aligned with `agents/_template/entrypoint.sh`.
- Agent-specific startup logic belongs in `/opt/agent/bin/start`.
- `/opt/agent/bin/start` is generated by the agent's `install.sh` during image build.
- `manifests/devbox.yaml.tmpl` should use `args: ["start"]` by default.

The shared entrypoint exports these standard variables:

- `AGENT_NAME`
- `AGENT_HOME=/opt/agent`
- `AGENT_START=/opt/agent/bin/start`
- `AGENT_DATA_DIR`
- `AGENT_WORKSPACE=/workspace`
- `AGENT_PORT`
- `AGENT_LOG_LEVEL`

## Image Versioning

GitHub Actions generates the build matrix from `registry/agents.yaml`.

Agent Hub deployment templates read the image from each agent's `template.yaml`:

```text
ghcr.io/<owner>/<agent>:latest
```

Release workflows also publish a traceable build tag:

```text
ghcr.io/<owner>/<agent>:build-YYYYMMDD-<12-char-sha>
```

After a successful release, the workflow keeps enabled agents' `template.yaml` image references on `:latest`.

## Agent Hub Templates

Each agent directory contains its Agent Hub deployment template:

- `template.yaml`: template metadata, access behavior, runtime settings, and model presets
- `manifests/devbox.yaml.tmpl`: Devbox Go template
- `manifests/service.yaml.tmpl`: Service Go template
- `manifests/ingress.yaml.tmpl`: Ingress Go template

The values in `template.yaml`, including `port`, `workingDir`, `user`, `defaultArgs`, `access`, and `manifestDir`, must match the resources rendered from `manifests/`.

## Local Validation

Run the contract validator before opening a PR:

```bash
bash test/validate-agent-contract.sh
```

Useful smoke tests:

```bash
bash test/hermes-smoke.sh
bash test/openclaw-smoke.sh
docker build -f agents/cowagent/Dockerfile -t agent-hub/cowagent:local .
```

Build the shared base image locally:

```bash
docker build -f base/Dockerfile -t agent-hub/agent-devbox-base:local .
IMAGE=agent-hub/agent-devbox-base:local bash base/smoke.sh
```
